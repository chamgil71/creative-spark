01

AI 출력을 검증하지 않고 사용

가장 흔하고 가장 위험한 함정

실제 사례

AI가 보고서에서 \"2025년 AI 시장 규모는 4.7조 달러\"라고 작성 → 실제: 4,700억 달러 (단위 오류, 10배 과장). 결재자가 발견하지 못하고 공식 보고서에 포함.

검증 원칙
수치·날짜 →

수치·날짜 → 반드시 1차 출처(SEC 공시, 공식 IR) 확인코드 라이브러리·메서드 → "이 메서드가 실제로 존재하는지 확인해줘"법령·정책 → law.go.kr 또는 공식 문서 대조기업 관계·제품 귀속 → 공식 웹사이트 직접 검색

02

과잉 신뢰와 책임 분산

"AI가 만들었으니 내 책임이 아니다" — 가장 위험한 태도

위험한 사고방식 vs 올바른 사고방식
❌ "AI가 만들었으니 내 책임은 아니다"
→ AI 출력물의 법적·도덕적 책임은 항상 사용자에게 있습니다
❌ "AI가 더 잘 알겠지"
→ 내 업무 맥락, 조직 문화, 법적 요건은 AI가 알 수 없습니다
❌ "AI가 틀릴 리 없어"
→ 최신 모델도 틀립니다. 한국어 도메인 전문 지식·수치 계산에서 특히
✅ AI = 빠른 초안 작성자
→ 내가 검토하고 수정하는 것이 최종 산출물
✅ AI = 지식 보조자
→ AI의 지식에는 시간 제한, 오류, 편향이 있음. 중요한 판단은 내 도메인 지식 우선
  • 금전이 관련된 결정 (예산, 계약, 지출 승인) — 사람 검토 필수
  • 개인정보 처리 (수집, 저장, 삭제) — 사람 검토 필수
  • 법적 효력이 있는 문서 (계약서, 공문, 신청서) — 사람 검토 필수
  • 외부 공개 콘텐츠 (보도자료, 공식 발표) — 사람 검토 필수
  • 운영 DB의 데이터 수정·삭제 — 사람 검토 필수

03

보안과 개인정보 노출

API 키 노출부터 개인정보 유출까지

실수 유형 발생 상황 예방법
API 키 GitHub 노출 .env 파일을 .gitignore에 미포함 프로젝트 시작 시 첫 번째로 .gitignore 설정
개인정보 AI 입력 이름·전화번호가 포함된 파일 첨부 데이터 익명화 후 입력 (홍길동 → 홍OO)
기밀 정보 공개 AI 입력 미공개 제품 로드맵, 미발표 실적 기밀 정보는 로컬 Ollama 모델 사용
service_role 키 노출 Supabase 키를 클라이언트 코드에 사용 service_role은 서버 전용, anon 키는 클라이언트
SQL Injection 취약 패턴 주의
❌ 취약한 패턴 (AI가 가끔 이렇게 생성)
query = f"SELECT * FROM users WHERE name = '{user_input}'"
→ user_input에 "'; DROP TABLE users; --" 입력 시 재앙
✅ 올바른 방법
query = "SELECT * FROM users WHERE name = ?"
cursor.execute(query, (user_input,))

04

맥락 소실과 일관성 붕괴

대화가 길어질수록 AI가 초반 규칙을 무시하기 시작

증상

20번 대화 후 AI가 httpx 대신 requests를 다시 사용, print()를 다시 사용, 폴더 구조를 바꿔버림.

🔄
🔄재정향 프롬프트
"[SKILL.md 재첨부]
방금 코드에서 SKILL.md 위반:
① requests 사용 → httpx로 교체 (규칙 §3)
② print() → logging으로 교체 (규칙 §3)
③ 타입 힌트 없음 → 모든 함수에 추가 (규칙 §2)
세 가지를 수정한 버전으로 다시 작성해줘."
💡
💡 예방:

매 대화 시작 시 SKILL.md 첨부 → Claude Projects에 등록 → 20회 이상 대화 후 새 대화창 시작. 이 세 가지로 맥락 소실을 대부분 예방할 수 있습니다.

05

버전 관리 없이 에이전트 실행

30개 파일 수정 후 원본이 없는 상황

사고 시나리오

Claude Code 자율 모드 실행 → "전체 리팩터링해줘" → AI가 30개 파일을 수정 → 결과가 마음에 안 들지만 원본이 없음 → 전체 프로젝트를 처음부터 다시 작성.

에이전트 실행 전 3중 안전망
# 1단계: Git 커밋 (반드시)
git add . && git commit -m "에이전트 실행 전 백업"
# 2단계: 원본 파일 불변
chmod 444 data/*.pdf # 원본 수정 금지
# 3단계: DB 스냅샷
cp budget.db budget.db.backup.$(date +%Y%m%d)
# 복구 시나리오
git checkout . # 코드 잘못됨 → 전체 되돌리기
git reset --hard HEAD~1 # 마지막 커밋 되돌리기

자동화 과잉 의존

자동화가 잘 될수록 생기는 역설적 위험

위험 1 — 블랙박스화

\"n8n이 알아서 해주니까\" → 실제 로직을 이해 못 함 → 오류 발생 시 원인 파악 불가.

위험 3 — 능력 퇴화

AI가 항상 대신 써주다 보면 스스로 글쓰기·분석 능력이 퇴화. AI는 \"초안 도구\"로만 활용, 최종 검토는 직접.

07

AI 생성 코드의 보안 취약점

AI가 기능적으로는 맞지만 보안상 위험한 코드를 생성하는 경우

🔍

코드 보안 검토 요청 패턴

코드 생성 후 항상 추가 요청:

"방금 생성한 코드에서 다음을 확인해줘:

□ SQL Injection 가능성

□ 하드코딩된 비밀번호나 API 키

□ 사용자 입력이 파일 경로에 직접 사용되는 부분

□ 민감 정보가 로그에 출력되는 부분

□ 외부 입력 검증 없이 처리하는 부분"

⚠
⚠️ 패스워드 하드코딩:

AI가 예시 코드에 PASSWORD = \"admin123\"처럼 실제처럼 보이는 값을 넣는 경우가 있습니다. 모든 비밀값은 반드시 환경 변수로 교체하세요.

올바른 AI 협업 마인드셋

AI는 강력한 도구입니다. 망치가 강력할수록 잘못 쓰면 더 큰 피해를 줍니다. 그러나 올바르게 사용하면 누구도 혼자서는 할 수 없는 일을 해냅니다.

AI의 역할
⚡
⚡ 빠른 초안 작성자

내가 검토하고 수정하는 것이 최종 산출물. AI가 만들었다는 이유만으로 그대로 사용하지 않는다.

AI의 역할
📚
📚 지식 보조자

AI의 지식에는 시간 제한·오류·편향이 있음. 중요한 판단은 내 도메인 지식과 경험이 우선.

AI의 역할
🔄
🔄 자동화 엔진

반복 작업을 없애주지만, 자동화된 오류도 반복된다. 검증 없는 자동화는 자동화된 실수다.

AI의 역할
🎓
🎓 학습 파트너

AI가 만든 코드를 읽으면서 새로운 패턴을 배울 수 있음. 수동적으로 받기만 하면 성장이 없다.

이 책을 마치며

이 책에서 다룬 모든 도구와 기법은 빠르게 발전하고 있습니다. 특정 도구의 사용법은 6개월 후 달라질 수 있습니다. 하지만 변하지 않는 것이 있습니다.

명확한 요구사항이 좋은 결과를 만든다

결과는 반드시 검증해야 한다

오류에서 배운 규칙이 쌓이면 실력이 된다

재사용 가능한 자산이 시간을 절약한다

AI는 도구이고, 책임은 사람에게 있다

💡

Key Takeaway · 마지막 장

AI 도구의 모든 함정은 결국 하나로 수렴합니다 —

"AI를 강력한 조수로 신뢰하되, 최종 검토와 책임은 항상 사람이 진다."

이 원칙 하나만 지켜도 충분합니다.
