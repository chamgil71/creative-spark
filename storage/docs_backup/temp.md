제공해주신 `guide-creation.md`와 `showcase.md` 파일을 바탕으로, **표준 키 정의와 쇼케이스 데이터 간의 정합성 및 혼선 여부**를 면밀히 검토했습니다.

정확한 근거와 명확한 데이터 매핑을 원하시는 만큼, 구조적인 불일치와 실전 활용 시 혼돈을 줄 수 있는 지점들을 구체적으로 짚어드리겠습니다.

## 🔍 종합 검토 의견

현재 구조에서 **가장 큰 혼선은 `meta` 키의 다중 정의**와 **일부 숏코드에서의 표준 키 이탈**에서 발생하고 있습니다.

개발자나 콘텐츠 작성자가 쇼케이스 파일(`showcase.md`)을 복사해서 쓸 때, "이 숏코드의 이 필드에는 무엇을 적어야 하는지" 헷갈릴 요소가 많습니다. 아래에서 핵심 문제점과 개선 방향을 정리해 드립니다.

## 🛠️ 주요 문제점 및 개선 피드백

### 1. `meta` 키의 정의 과부하 (Overloading) 및 혼선

표준 키 설명에서는 `meta: 부가 데이터 (도구명·파이프| 구분 목록)`라고 정의되어 있지만, 쇼케이스에서는 이 하나의 키가 너무나도 다른 포맷과 의미로 사용되고 있습니다.

* **포맷 불일치:** * `workflow`에서는 단일 텍스트(`meta: Claude AI`)로 사용됩니다.
* `plan-grid`, `compare-2col`, `bottom-list`에서는 파이프 분리 문자열(`meta: 기능A|기능B|기능C`)로 사용됩니다.


* **의미 혼선:** `plan-grid`에서는 '제공 기능 목록'이지만, `compare-2col`에서는 '비교 내용(문장)', `bottom-list`에서는 '하단 칩(키워드)'으로 작동합니다.
* 👉 **피드백:** 개발 프로세스상 파서(Parser)가 이를 분리하는 로직을 가질 텐데, 문장형 파이프와 키워드형 파이프를 `meta` 하나로 처리하면 마크다운 작성자가 실수하기 쉽습니다. 가이드 문서에 **"어떤 숏코드에서 `meta`가 배열(파이프)로 작동하는지"** 명시하거나 키 이름을 분리(`items` 또는 `chips`)하는 것이 안전합니다.

### 2. 쇼케이스 문법 오류 (빌드 에러 유발 항목)

`showcase.md` 내부의 **10. 알림 박스 (alert-box)** 예시에서 치명적인 YAML 문법 오류가 발견되었습니다.

```md
::: alert-box
- type: tip
  title: 💡 핵심 팁
  desc: ...

  - type: warn  # ← 잘못된 들여쓰기 및 대시(-) 중복
  title: ⚠️ 주의사항
  desc: ...
:::

```

* 👉 **피드백:** `guide-creation.md`에서 *"잘못된 YAML frontmatter는 오류를 유발합니다. 들여쓰기를 확인하세요."*라고 경고했으나, 정작 쇼케이스 본문 숏코드 내에서 들여쓰기 오류가 있습니다. `::: alert-box` 내부에 여러 개를 넣으려면 대시(`-`)의 시작 라인을 정확히 맞춰야 합니다.

### 3. 표준 6대 키 범위를 벗어난 '미정의 키' 사용

표준 키(6대 키: `icon`, `title`, `desc`, `tag`, `meta`, `note`, `color`)에 등록되지 않은 키들이 쇼케이스에 다수 등장합니다.

* `plan-grid`: **`featured: "true"`** 사용됨.
* `alert-box`: **`type: tip`** 사용됨.
* `command-block` (가이드에만 있고 쇼케이스엔 누락): **`label`** 사용됨.
* 👉 **피드백:** 이 키들은 해당 숏코드의 렌더링에 필수적인 '특수 키'입니다. 표준 키 테이블 아래에 [숏코드별 전용 특수 키]라는 항목을 신설하여 `featured`, `type`, `label`을 명시해 주어야 작성자가 혼란을 겪지 않습니다.

## 💡 최종 개선 제안 (Action Item)

작성자의 혼선을 줄이고 파이프라인의 완성도를 높이기 위해 두 파일의 내용을 아래와 같이 동기화하는 것을 추천합니다.

1. **`guide-creation.md` 내부의 숏코드 설명 보완**
`meta`라는 이름이 너무 추상적이므로, 가이드 문서의 숏코드 목록 설명란에 파이프(`|`) 활용법을 더 명확히 적어주세요.
* 예: `compare-2col` 설명 -> `meta (비교 항목들을 '|'로 구분하여 작성)`


2. **`showcase.md` 내 알림 박스 문법 수정**
작성자가 복사해서 바로 쓸 수 있도록 들여쓰기를 올바르게 교정해야 합니다.
3. **특수 키 정의 추가**
`type`(alert-box용), `featured`(plan-grid용), `label`(command-block용)을 가이드 문서의 '6대 표준 키' 아래에 '숏코드별 예약 키'로 별도 명시해 주면 혼선이 완벽히 해결될 것입니다.



제안해주신 "특수 키들을 다른 표준 키로 통합하는 방안"은 시스템의 복잡도를 낮추고 작성자의 혼선을 줄이는 데 **매우 좋은 접근 전환**입니다.

결론부터 말씀드리면, **`label`과 `featured`는 기존 표준 키로 완벽히 통합이 가능**하지만, **`type`은 성격상 표준 키로 통합 시 오히려 데이터 오염(혼선)을 유발할 수 있어 제한적으로 분리하거나 다른 방식으로 치환**하는 것이 좋습니다.

구체적으로 어떻게 통합하고 정리를 유도할 수 있을지, 각 키별 타당성과 대안을 명확한 근거와 함께 분석해 드리겠습니다.

---

### 1. `label` (command-block용) ➡️ `tag` 또는 `title`로 통합 (**적극 권장**)

* **현황:** 가이드 문서상 `command-block`은 파일 복사나 터미널 명령어 라벨을 표시하는 용도입니다.
* **통합 제안:** 이를 표준 키인 **`tag`** 또는 `title`로 대체합니다.
* **타당성 근거:** 명령어 블록의 제목이나 라벨은 성격상 '메타 정보'나 '태그/타이틀'과 완벽히 일치합니다.
```yaml
# 변경 전 (특수 키 사용)
- label: "종합 빌드"
  desc: "npm run build"

# 변경 후 (표준 키로 통합)
- tag: "종합 빌드"  # 또는 title: "종합 빌드"
  desc: "npm run build"

```


* **결론:** 시스템 파서(Parser)에서 `label` 대신 `tag`를 읽도록 수정하면 되므로 **통합하는 것이 훨씬 직관적**입니다.

---

### 2. `featured` (plan-grid용) ➡️ `tag`로 통합 (**적극 권장**)

* **현황:** 요금제나 그리드에서 특정 항목을 강조(추천 상품 등)하기 위해 `featured: "true"` 형태로 사용 중입니다.
* **통합 제안:** 이를 표준 키인 `tag`에 특정 키워드(예: `추천`, `인기`, `Featured`)를 넣는 방식으로 통합합니다.
* **타당성 근거:** 굳이 불리언(`true`/`false`) 전용 키를 만드는 것보다, `tag: "추천"`이 존재할 때 디자인적으로 강조 스타일(배경색 변경 등)이 들어가도록 파서 로직을 짜는 것이 마크다운 친화적입니다.
```yaml
# 변경 전 (특수 키 사용)
- title: "Pro 플랜"
  featured: "true"

# 변경 후 (표준 키로 통합)
- title: "Pro 플랜"
  tag: "추천"  # 파서가 tag == "추천"일 때 강조 디자인 적용

```


* **결론:** 작성자 입장에서도 새로운 키를 외울 필요가 없어 **통합을 강력히 추천**합니다.

---

### 3. `type` (alert-box용) ➡️ 통합 시 주의 필요 (**조건부 대안 제안**)

* **현황:** 알림창의 종류(`tip`, `warn`, `success`, `danger`)를 결정하는 제어용 키입니다.
* **통합 시 문제점:** 이를 만약 `tag`나 `title`로 통합해 버리면, 작성자가 실수로 `tag: "팁"` 대신 `tag: "꿀팁"`이라고 썼을 때 시스템이 이를 인식하지 못하고 스타일 렌더링이 깨지는 문제가 발생합니다. 즉, `type`은 단순 텍스트가 아니라 시스템의 '예약어(스타일 제어 코드)'입니다.
* **대안 제안 (통합 대신 '문법적 전환'):** `alert-box` 내부의 YAML 키로 `type`을 남겨두는 대신, **마크다운 자체의 클래스 문법을 활용**하는 방향으로 전환하면 혼선을 완전히 없앨 수 있습니다.
```md
# 대안: YAML 내부 키를 없애고 숏코드 선언부에 녹여내기
::: alert-box tip
- title: 💡 핵심 팁
  desc: shortcode-map.json을 수정하면...
:::

::: alert-box warn
- title: ⚠️ 주의사항
  desc: 잘못된 YAML frontmatter는...
:::

```


* **결론:** 이렇게 바꾸면 개별 데이터 아이템(`-`) 내부에는 오직 표준 키(`title`, `desc`)만 남길 수 있어서 **완벽한 키 통합 효과**를 볼 수 있습니다.

---

### 📋 요약 및 제안하는 최종 표준 키 매핑 구조

제안하신 대로 키를 통합하고 정돈하면, 가이드 문서의 '표준 키' 섹션을 아래와 같이 깔끔하게 단일화할 수 있습니다. 특수 키라는 개념 자체를 없앨 수 있습니다.

* **`title`**: 제목 / 명령어 라벨 (`command-block` 통합)
* **`desc`**: 상세 설명 / 실제 명령어 본문
* **`tag`**: 상단 칩 / 강조 표시 (`plan-grid`의 `featured` 통합)
* **`meta`**: 부가 데이터 (도구명 또는 파이프`|` 구분 목록)
* **`icon`, `note`, `color**`: 기존과 동일
* **`alert-box`**: 상단 숏코드 선언문 자체에서 `::: alert-box tip` 형태로 타입을 지정하도록 파서 변경

이렇게 다른 표준 키의 역할을 조금씩 확장하여 통합하는 방향으로 가이드와 파서 시스템을 개편하시는 것을 적극 추천합니다. 작성의 편의성과 데이터 일관성이 훨씬 높아질 것입니다.
