일반인 입문 로드맵

🧹
시작 전에 버려야 할 오해 3가지
❌ 오해

"코딩을 먼저 배워야 AI 도구를 쓸 수 있다"

✅ 사실

AI 도구를 쓰면서 필요한 개념이 자연스럽게 따라옵니다. "무엇을 만들고 싶은지"가 먼저입니다.

❌ 오해

"AI가 다 해주니까 결과를 그냥 쓰면 된다"

✅ 사실

AI는 강력한 조수이지 책임자가 아닙니다. 결과를 검토하고 판단하는 것은 항상 사람의 몫입니다.

❌ 오해

"유료 플랜을 써야 제대로 할 수 있다"

✅ 사실

Claude.ai 무료 · Lovable 무료 · Vercel 무료로 상당한 수준의 결과물을 만들 수 있습니다.

STEP 1 · 1주차
🤖 첫 코드 받기
AI와 대화해서 실제로 동작하는 코드를 처음으로 받아보는 단계. 도구: Claude.ai (무료)
  • Claude.ai에서 "Excel 파일의 숫자를 모두 더하는 Python 코드" 요청해보기
  • 결과를 보고 "이게 맞나?" 판단하는 연습
  • 오류 메시지를 Claude에게 붙여넣어 수정 요청하기
  • 오류를 만나고 → 물어보고 → 해결하는 사이클 3회 경험
💡 1주차 목표: 코드를 완벽하게 만드는 것이 아니라, 오류와 친해지는 것. v15까지 간 파이프라인도 처음은 오류였습니다.
STEP 2 · 2~3주차
📱 첫 앱 만들기
코드 없이 동작하는 웹 앱을 만들고 URL로 공유. 도구: Lovable (무료) + GitHub + Vercel
  • Lovable에서 "팀 주간 업무 보고 수집 앱" 자연어로 요청
  • 약 3~5분 후 동작하는 앱과 URL 자동 생성 → 팀원에게 공유
  • GitHub 계정 만들기 → Lovable에서 "Connect to GitHub"
  • vercel.com에서 해당 레포 Import → https://내앱.vercel.app 자동 생성
💡 핵심 이해: 코드 → GitHub (저장) → Vercel (공개). 이 흐름이 현대 웹 서비스 배포의 기본입니다.
STEP 3 · 1~2개월차
⚡ 첫 자동화
사람이 하던 반복 작업 하나를 완전히 없애기. 도구: n8n + Slack
  • n8n Schedule 트리거 → 날씨 API → Slack 메시지 전송 (가장 단순한 자동화)
  • RSS 수집 → Ollama 3줄 요약 → Slack 전송 (AI 포함 자동화)
  • n8n 실행 로그 보고 원인 파악하기
  • "이 자동화 덕분에 안 해도 되게 된 작업" 1가지 이상 달성
💡 시작 추천: "매일 오전 9시에 날씨 정보를 Slack에 전송". 트리거·처리·액션의 3노드 구조를 이해하면 모든 자동화의 기반 완성.
STEP 4 · 3개월 이후
🏗️ 시스템 구축
여러 도구가 연결된 나만의 자동화 시스템 운영. 단계 1~3은 도구를 익히는 과정, 단계 4는 그 도구들이 시스템으로 동작하기 시작하는 단계입니다.
  • n8n이 공고를 수집하고 → Claude가 분류하고 → Notion에 저장되고 → Slack 알림
  • SKILL.md로 AI 작업 규칙 정리 → Obsidian으로 지식 축적 → GitHub 버전 관리
  • 나의 업무에서 가장 큰 시간을 절약할 파이프라인 하나 완성

자주 묻는 질문

Python을 먼저 배워야 하나요?
필수는 아닙니다. 하지만 Python 기초(변수, 반복문, 함수)만 이해해도 Claude가 생성한 코드를 읽고 판단할 수 있습니다. "AI가 만든 코드를 이해하는 수준"이 현실적인 목표입니다.
유료 플랜이 필요한가요?
처음 한 달은 전부 무료로 시작할 수 있습니다. Claude.ai 무료 · Lovable 무료 · Vercel 무료 · n8n 셀프호스팅 무료. 한계를 느끼면 그때 유료를 검토하세요.
회사 데이터를 AI에 넣어도 되나요?
개인정보, 미공개 기밀, 고객 데이터는 AI에 입력하지 마세요. 데이터를 익명화하거나 구조만 설명하는 방식으로 우회하면 대부분의 작업이 가능합니다. 조직의 AI 활용 정책을 먼저 확인하세요.
어느 정도면 "잘 한다"고 할 수 있나요?
다음 3가지를 할 수 있으면 충분합니다: ① 내 업무의 반복 작업 하나를 자동화하기, ② AI가 만든 결과를 검토하고 수정 요청하기, ③ 잘 된 방법을 다음에도 쓸 수 있게 기록해두기.
💡
Key Takeaway
입문 로드맵의 핵심은 1주차 첫 코드 → 3주차 첫 앱 → 2개월차 첫 자동화 순서로 작은 성공을 쌓아가는 것. 완벽한 시스템보다 "이 반복 작업 하나를 없애기"가 시작점입니다.

좋은 프롬프트 쓰는 법

AI에게 잘 말하는 것은 기술입니다. 그 기술은 배울 수 있습니다. 같은 목적이라도 어떻게 말하느냐에 따라 AI의 결과물 품질이 크게 달라집니다.

품질 차이가 나는 이유
── 나쁜 프롬프트 ──────────────────────
"보고서 요약해줘"

── 좋은 프롬프트 ──────────────────────
"다음 AI 정책 보고서를 요약해줘.
독자: 비전문가 정책 담당자 (AI 지식 없음)
분량: 5문장 이내
필수 포함: 핵심 정책 목표, 예산 규모, 기대 효과
문체: 격식체, 명사형 종결
금지: 기술 용어, 약어 (풀어서 설명)"

핵심 기법 10가지

01
역할 부여 (Role Prompting)

"너는 10년 경력의 Python 성능 최적화 전문가야." — 역할을 부여하면 해당 도메인의 관점과 용어로 더 전문적인 답변이 나옵니다.

02
입출력 명세 (I/O Specification)

"입력: CSV (컬럼: 날짜·부서명·금액) / 출력: JSON {department, monthly_total: {YYYY-MM: N}}" — 구조가 명확할수록 결과가 정확합니다.

03
예시 제공 (Few-Shot Prompting)

원하는 출력 형식의 예시를 1~3개 포함하면 AI가 패턴을 파악합니다. "예시 입력 → 예시 출력" 형식으로 제공하세요.

04
단계별 사고 유도 (Chain of Thought)

"1. 각 부서 상황 파악 → 2. 제약 조건 확인 → 3. 배분 원칙 적용 → 4. 최종안 도출" — 복잡한 분석은 단계를 명시하면 정확도가 높아집니다.

05
부정 지시 (Negative Constraints)

"❌ 금지: '다양한', '여러', '등' 같은 모호한 표현 / ❌ 금지: 글머리표 없는 산문 형식" — 원하지 않는 것을 명시하면 AI가 그 방향을 피합니다.

06
출력 형식 고정 (Format Locking)

자동화 파이프라인에서 AI 출력을 파싱해야 할 때: "반드시 다음 JSON 형식으로만 출력해줘. 다른 설명이나 마크다운 없이 JSON만."

07
컨텍스트 최소화 (Context Scoping)

3페이지 배경 설명보다 현재 작업에 필요한 맥락만 정확히 제공하는 것이 효과적입니다. 필요한 것만 → 핵심만 → 간결하게.

08
확인 요청 (Verification Request)

"코드 작성 전에 확인: □ 존재하지 않는 메서드 없음? □ None 처리 완료? □ SKILL.md 규칙 준수?" — 스스로 검토를 요청하면 오류가 줄어듭니다.

09
반복 개선 (Iterative Refinement)

완벽한 프롬프트를 처음부터 만들려 하지 않습니다. 빠르게 결과를 받고, 부족한 부분을 피드백하며 3라운드 개선이 첫 완벽한 프롬프트보다 낫습니다.

10
메타 프롬프트 (Meta Prompting)

"내 초안: '병합 셀 처리 코드 만들어줘'. 이 프롬프트를 더 구체적이고 효과적으로 개선해줘." — AI에게 더 좋은 프롬프트를 만들게 합니다.

흔한 실수 10가지

MISTAKE 01
목표 없는 질문
"이 코드 좋아?" — 목표가 없어 AI가 무엇을 평가해야 할지 모릅니다.
→ "이 코드에서 성능 개선 가능한 부분을 찾아줘"
MISTAKE 02
형식 미지정
"장단점 알려줘" — AI가 임의의 형식으로 출력합니다.
→ "장점 3가지, 단점 3가지를 표로 정리해줘"
MISTAKE 03
한 번에 너무 많이
"파싱하고 정제하고 저장하고 보고서도 만들어줘" — 복잡도 과부하.
→ 작업을 분리해서 순서대로 요청
MISTAKE 04
수정 시 원본 미제시
"아까 그 코드에서 None 처리 추가해줘" — 대화가 길어지면 AI가 맥락을 잃습니다.
→ 코드 다시 붙여넣기 + "None 처리를 추가해줘"
MISTAKE 05
부정적 피드백만
"이거 틀렸어" — 어디가 왜 틀렸는지 AI가 모릅니다.
→ "3번 함수에서 예상: 1234567, 실제: 2469134"
MISTAKE 06
출력 형식 미지정 (자동화 시)
"중요도를 분류해줘" — 파이프라인에서 파싱 불가.
→ "'상', '중', '하' 세 값 중 하나로만 반환"
MISTAKE 07
검증 요청 생략
AI 환각으로 존재하지 않는 함수가 코드에 포함됩니다.
→ "존재하지 않는 함수 있으면 알려줘" 항상 추가
MISTAKE 08
SKILL.md 재첨부 생략
20번 이상 대화 후 AI가 초반 규칙을 무시하기 시작합니다.
→ 긴 대화 후에는 SKILL.md를 다시 첨부
MISTAKE 09
AI 출력 무검증
수치, 날짜, 고유명사를 그대로 신뢰합니다.
→ 수치·날짜·고유명사는 반드시 원본 확인
MISTAKE 10
프롬프트 버리기
잘 작동한 프롬프트를 기록하지 않아 다음에 재발명합니다.
→ Obsidian 01-Prompts/에 즉시 저장

상황별 프롬프트 패턴

상황핵심 요소예시 지시
코드 생성 언어·입출력 명세·라이브러리·금지 패턴 "타입 힌트, 한국어 docstring, 방어 코드 포함. httpx 사용 (requests 금지)"
문서 작성 독자·분량·구조·문체·금지 표현 "비전문가 관리자 대상, A4 5페이지, □○― 구조, 명사형 종결"
데이터 분석 분석 관점·출력 형식·주의사항 "전월 대비 변화에 초점, 수치 기반 인사이트 3가지, 상관≠인과"
자동화 파이프라인 출력 형식 고정·오류 처리 방식 "JSON만 출력, 파싱 실패 시 null, '상/중/하' 세 값만"
번역·문체 변환 목표 독자·유지 요소·금지 요소 "수치·고유명사 그대로, 어색한 직역 금지, 격식체"
💡 프롬프트 아카이빙: 잘 작동한 프롬프트는 Obsidian의 01-Prompts/ 폴더에 태그·성공률·관련 노트와 함께 저장하세요. 검증된 프롬프트 50개가 쌓이면 새 프로젝트 시작 속도가 2배 이상 빨라집니다.
💡
Key Takeaway
좋은 프롬프트는 역할·입출력 명세·예시·부정 지시·출력 형식 고정 다섯 가지를 갖추며, 완벽한 첫 번째 프롬프트보다 빠른 피드백으로 반복 개선하는 것이 더 효과적입니다.
