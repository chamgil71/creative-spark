# PPT Design System — 구현 계획서 (PRD)

**버전**: v1.0
**작성일**: 2026-06-17
**상태**: 인터뷰 기반 1차 확정안 — 구현 착수 전 검토용

---

## 1. 개요

### 1.1 배경
NIPA 등 외부 기관이 배포하는 PPT 템플릿에 맞춰, 보유 중인 다수의 PPT 파일을 디자인 표준(색상·폰트·위치)으로 일괄 변환해야 하는 반복 작업이 존재한다. 기존에 구축한 `pptx_inspector.py`/`pptx_apply.py` 라운드트립 툴체인(Python)이 있으나, 이번 시스템은 **백엔드 없는 React/Vite 단일 페이지 애플리케이션**으로 별도 설계한다.

### 1.2 목적
target PPT 파일을 업로드하면 색상·도형·위치·폰트 정보를 분석하고, raw 템플릿(기준 디자인) 기준으로 일괄 변경할 항목과 유지할 항목을 항목 단위로 선택해 수정된 PPTX를 즉시 다운로드받는다.

### 1.3 범위 (Scope)
| 구분 | 포함 여부 |
|---|---|
| 색상 분석 + 일괄 변경 (테마 역할 단위 + 비표준 RGB 단위) | 1차 포함 |
| 폰트 분석 + 일괄 변경 (패밀리/크기/굵기/색상) | 1차 포함 |
| 위치·크기 분석 + 수동 확인 후 적용 (타이틀/챕터명/본문범위/페이지번호) | 1차 포함 |
| 도형(그라데이션/그림자/입체/반사/윤곽) 분석 | 1차 포함 (분석만, 쓰기 제외) |
| 도형 효과 실제 적용(쓰기) | 2차 보류 |
| 이미지 교체 | 2차 보류 |
| 슬라이드 페이지 삽입 | 2차 보류 |
| 마스터·레이아웃·테마 등록 | 2차 보류 |

## 2. 사용자 워크플로우

```
[1] raw 템플릿 업로드
     └→ 색상/도형/위치/폰트 자동 분석
     └→ "기본 설정"으로 저장 (React state, 필요시 JSON export)
     └→ GUI에서 기본 설정값 미세조정 가능 (편집 화면)

[2] target PPTX 업로드
     └→ PPT 분석 (색상/도형/위치/폰트 정보 추출)
     └→ 분석 결과 화면 표시 + JSON 다운로드 가능 (+ JSON 업로드로 재편집 가능)

[3] 탭/분기별 수정 화면
     ├ 색상 탭: 테마색 역할별 + 비표준 RGB별 "유지/변경" 개별 체크
     ├ 폰트 탭: 텍스트 역할별(제목/본문 등) 패밀리·크기·굵기·색상 개별 체크
     ├ 위치 탭: 기본 설정파일의 프리셋 좌표 적용 + 항목별 세부 직접 조정
     └ 도형 탭: 그라데이션/그림자/입체/반사/윤곽 적용 여부 "확인만" (읽기 전용 표시)

[4] 적용 실행
     └→ 백업(원본 보존) → 변경 적용 → 결과 검증(diff) → 실패 시 롤백
     └→ 수정된 target PPTX 다운로드
```

## 3. 시스템 아키텍처

### 3.1 기술 스택 (확정)
- **프론트엔드**: React + Vite (TypeScript)
- **백엔드**: 없음 — 전 과정 클라이언트 사이드 처리
- **PPTX 파싱/조작**: `JSZip`(ZIP 압축/해제) + XML 파서(`fast-xml-parser` 또는 `DOMParser`)
- **배포**: GitHub Pages 또는 Vercel (정적 호스팅, 빌드 산출물만 배포)

### 3.2 백엔드 미사용 결정에 대한 근거
PPTX는 ZIP 컨테이너 안에 OOXML(XML) 파일들이 들어있는 구조이므로, `JSZip`으로 압축 해제 → XML 파싱/수정 → 재압축하는 전체 파이프라인이 브라우저 내에서 동작 가능하다. 따라서 별도 서버(FastAPI 등)나 Supabase 같은 DB는 필요하지 않다. Supabase 등 외부 저장소가 필요해지는 시점은 ① 여러 사람/기기 간 템플릿 설정 공유, ② 사용자 인증이 필요할 때뿐이며, 현재 범위에서는 해당하지 않는다.

### 3.3 기존 Python 자산과의 관계
`pptx_inspector.py`/`pptx_apply.py`의 로직을 1:1 포팅하지 않는다. 검증된 XML 접근 패턴(어떤 태그를 어떻게 다뤄야 하는지)은 참고하되, JS/TS 생태계 구조에 맞게 처음부터 재설계한다. Python 버전은 별도 자산으로 유지한다.

### 3.4 데이터 보관 방식
- 작업 중에는 React state로만 보관 (세션 한정 기본 동작)
- 사용자가 원하면 분석/설정 결과를 JSON으로 export/import 가능 (작업 중단 후 재개 지원용 옵션)
- 영구 DB 저장은 1차 범위 제외

## 4. 기능 요구사항

### 4.1 환경설정 (raw 템플릿 분석)
- raw 템플릿(.pptx) 업로드 시 색상/도형/위치/폰트를 자동 분석해 "기본 설정"으로 저장
- 기본 설정은 GUI 편집 화면에서 직접 수정 가능 (색상값, 도형 효과 확인, 위치 좌표, 폰트)
- 기본 설정은 JSON으로 다운로드/재업로드 가능

### 4.2 target PPTX 분석
분석 항목과 단위:
| 항목 | 분석 단위 |
|---|---|
| 색상 | 테마 색상 역할(강조색1~6, 배경, 텍스트 등) + 테마에 없는 비표준 RGB 개별 값 |
| 폰트 | 텍스트 역할(제목/부제목/본문 등)별 패밀리, 크기, 굵기, 색상 |
| 위치/크기 | 타이틀, 챕터명(부제목), 본문 범위, 페이지번호의 좌표(x, y, width, height) |
| 도형 | 그라데이션, 그림자, 입체, 반사, 윤곽 적용 여부 및 속성값 (읽기 전용) |

분석 결과는 화면에 표시되며, 항목별 JSON 다운로드 및 재업로드(편집 후 반영)를 지원한다.

### 4.3 매칭 전략 (raw 템플릿 ↔ target PPTX)
1. **1차 기준**: placeholder 타입(title, body, slide number 등) 기준 매칭
2. **보조 기준**: placeholder 타입이 없거나 구조가 표준에서 벗어난 도형은 이름(shape name)·순서(z-order) 휴리스틱으로 보조 매칭
3. 매칭 결과는 사용자가 확인할 수 있도록 화면에 노출 (자동 매칭 오류 가능성을 사용자가 직접 검수)

### 4.4 수정 적용 로직
- **색상/폰트**: 항목 단위(개별 체크박스)로 "유지" 또는 "raw 템플릿 기준으로 변경" 선택. 기본값은 "변경"이며 항목별 예외 처리 가능
- **위치/크기**: 기본 설정파일의 프리셋 좌표를 1차 적용한 뒤, 화면에서 세부 직접 조정 가능. 자동 일괄 적용이 아니라 적용 전 수동 확인 단계를 거침
- **도형**: 1차 버전에서는 적용(쓰기) 기능 없음. 분석 결과만 표시

### 4.5 안전장치 (쓰기 작업 대상: 색상, 폰트, 위치/크기)
- 적용 전 원본 파일 백업 보존
- 적용 후 결과 diff 검증
- 검증 실패 시 자동 롤백 (원본 손상 방지 최우선)
- 도형 효과 등 2차 범위의 향후 쓰기 작업에는 해당 안전장치 수준을 적용하지 않고 추후 별도 검토

## 5. 비기능 요구사항

### 5.1 성능
- 1차 버전은 성능 최적화(Web Worker 분리 등)를 별도로 설계하지 않는다
- 대용량 파일(이미지·효과가 많은 슬라이드) 처리 시 느려지거나 멈추는 문제는 발생 시점에 대응 (사전 최적화는 2차)
- **인지된 최대 리스크**: 브라우저 메모리 내 대용량 PPTX(다수 이미지·효과) 처리 시 성능 저하 또는 응답 없음

### 5.2 배포
- GitHub Pages 또는 Vercel을 통한 정적 배포
- 빌드 산출물만 호스팅, 런타임 서버 코드 없음

## 6. 리스크 및 완화책

| 리스크 | 영향 | 완화책 |
|---|---|---|
| 대용량 PPTX 처리 시 브라우저 성능 저하 (24-A 직접 지적) | 1차 가장 가능성 높은 실패 원인 | 1차는 미대응, 발생 시 파일 크기 경고 → Web Worker 분리 순서로 단계적 대응 |
| placeholder 구조가 표준과 다른 target 파일에서 매칭 오류 | 잘못된 위치에 색상/폰트 적용 | 매칭 결과를 사용자에게 노출해 수동 검수 단계 확보 |
| Python 로직을 포팅하지 않고 재설계 | 기존에 겪은 XML 처리 버그(예: linesegarray류)를 PPTX에서 재경험할 가능성 | 핵심 XML 조작 단계마다 백업+diff 검증 의무화 |
| 효과(그라데이션 등) XML 구조가 OOXML 표준상 복잡 | 2차 범위 진입 시 절반만 구현될 위험 (24-B로 사전 인지) | 1차는 분석(읽기)까지만 범위 한정, 쓰기는 2차 별도 스파이크 진행 |

## 7. 단계별 마일스톤

### Phase 1 (MVP, 이번 구현 범위)
1. raw 템플릿 업로드 → 색상/폰트/위치/도형 자동 분석 → 기본 설정 저장
2. target PPTX 업로드 → 동일 항목 분석 → 화면 표시 + JSON export/import
3. 매칭 로직 (placeholder 타입 기준 + 이름/순서 보조)
4. 색상/폰트 탭: 항목별 유지/변경 체크 → 적용 → 백업/diff/롤백
5. 위치 탭: 프리셋 + 세부조정 → 적용 (수동 확인 단계 포함)
6. 도형 탭: 분석 결과 읽기 전용 표시
7. 수정된 target PPTX 다운로드

### Phase 2 (추가 과제, 보류)
- 도형 효과 실제 쓰기 적용
- 특정 이미지 교체
- 특정 슬라이드 페이지 삽입
- 마스터·레이아웃·테마로 등록

## 8. 미해결 질문 (Open Issues)

- JSON export/import 시 스키마 버전 관리 방식 (포맷이 바뀔 경우 하위 호환)
- 매칭 휴리스틱(이름/순서 기반)의 구체적 알고리즘 — 어떤 이름 패턴/순서 규칙을 쓸지는 구현 중 결정 필요
- 대용량 파일 임계치(몇 MB, 몇 슬라이드부터 "대용량"으로 간주하는지)는 미정 — Phase 1 운영 중 실측 후 결정

## 9. 다음 단계 (즉시 행동)

1. PPTX OOXML 구조 중 색상(테마 `theme1.xml`)·폰트(`a:rPr`)·위치(`a:off`, `a:ext`) 관련 XML 스키마 사전 조사
2. `JSZip` + `fast-xml-parser` 조합으로 PPTX 압축해제→파싱→재압축 최소 PoC 작성 (읽기만)
3. React/Vite 프로젝트 스캐폴딩 + 탭 구조(색상/폰트/위치/도형) 화면 와이어프레임
4. 매칭 로직 1차 구현 후 raw 템플릿 + target PPTX 샘플로 매칭 정확도 검증
