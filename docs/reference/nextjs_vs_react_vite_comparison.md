# Next.js vs React + Vite 아키텍처 비교 및 기술 스택 선정 이유

본 문서는 프로젝트의 웹 프레임워크 기술 스택 선정 과정에서 검토된 Next.js와 React+Vite의 본질적인 기술 차이점, 그리고 본 프로젝트의 유니크한 제약 조건 하에서 React+Vite가 채택되어야 하는 객관적인 기술적 근거를 설명합니다.

---

## 1. 두 기술의 본질적 아키텍처 차이

| 비교 항목 | React + Vite (CSR 기반 SPA) | Next.js (풀스택 웹 프레임워크) |
| :--- | :--- | :--- |
| **렌더링 패러다임** | **Client-Side Rendering (CSR)**<br>- 최초 진입 시 빈 HTML과 대용량 JS 번들을 로드한 후, 클라이언트 브라우저 상에서 리액트 자바스크립트가 DOM을 동적으로 구성. | **Server-Side Rendering (SSR) / SSG**<br>- 클라이언트의 요청이 도달하면 Node.js 서버 환경에서 HTML 마크업을 동적으로 프리렌더링하여 클라이언트로 즉시 전송. |
| **인프라/서버 요구사항** | **정적 웹서버만 필요 (Serverless)**<br>- 별도의 백엔드 가동 서버 없이 AWS S3, GitHub Pages 또는 로컬 PC 디렉토리 자체에서 HTML 더블클릭 기동 가능. | **Node.js 런타임 서버 필수 (Server-bound)**<br>- dynamic route 및 서버 컴포넌트 처리를 위해 실시간 Node.js 웹 애플리케이션 서버가 배포 구동되어야 함. |
| **검색엔진 최적화 (SEO)** | **비교적 취약**<br>- 최초 HTML 바디가 비어 있어 크롤링 봇이 자바스크립트를 정상 실행하기 전까지는 노출에 제한이 있음. | **매우 우수**<br>- 서버 측에서 이미 텍스트 정보가 담긴 정적 HTML을 서빙하므로 포털 노출 및 크롤링에 극히 유리. |
| **이식성 및 독립성** | **매우 높음**<br>- 1파일 오프라인 standalone 번들링 가능. | **낮음**<br>- 번들 파일이 여러 정적 리소스로 미세 split되어 서버의 라우팅 매핑 테이블 없이는 구동 불가능. |

---

## 2. 본 프로젝트의 스택 선정 의사결정 (React + Vite 선정 이유)

본 프로젝트는 단순한 정보 서빙 웹사이트가 아닌, 마크다운 가이드북을 다양한 슬라이드와 바이너리 PPTX로 교차 컴파일하는 **"로컬 에뮬레이터 에디터 및 개발 도구(Utility IDE)"**의 성격을 가집니다. 따라서 다음 3가지 핵심 제약으로 인해 **React + Vite** 스택이 절대적으로 유리합니다.

### 💡 의사결정 근거 1: 오프라인 Standalone (단독 무설치 1파일) 지원
* **요구사항**: 망 격리 환경(보안망) 및 로컬 웹 서버가 없는 환경에서도 `dist/standalone.html` 파일 더블클릭만으로 가이드북 허브 전체를 열람하고 구동할 수 있어야 함.
* **적합성**: React + Vite 번들러는 CSS, JS, 그리고 빌드된 JSON 가이드북 데이터셋을 단 하나의 정적 HTML 파일 구조 내부로 병합(inlining)하는 컴파일 파이프라인 구성이 지극히 용이함. 반면 Next.js는 정적 컴파일(`next export`)을 수행하더라도 여러 조각의 번들로 라우팅이 쪼개지므로, 1파일 임베딩 아키텍처가 완전히 무력화됨.

### 💡 의사결정 근거 2: SEO(검색엔진 최적화) 무관성
* **요구사항**: 본 도구는 일반 포털 검색 노출을 위한 블로그나 이커머스가 아님.
* **적합성**: SEO 점수가 프로젝트의 가치에 기여하지 않으므로 Next.js의 고비용 SSR 인프라를 유치할 명분이 없으며, 오히려 브라우저 상에서 리액트 가상 DOM을 활용해 나누어진 Pane을 리사이징하고 테마를 즉석에서 렌더링하는 클라이언트 반응성이 극대화되어야 하므로 CSR(React + Vite)이 적합함.

### 💡 의사결정 근거 3: 빌드 경량화 및 개발자 생산성
* **적합성**: Vite는 HMR(Hot Module Replacement) 속도가 현존 빌드 도구 중 가장 빠르며, 빌드 산출물의 구조가 단순하여 `vite-api-plugin.ts` 미들웨어와의 바인딩이 매우 직관적임. Next.js 도입 시 불필요하게 늘어나는 번들 용량 및 라우팅 오버헤드를 막고 가벼운 파일 입출력 스택을 유지할 수 있음.

---

## 3. 결론

본 프로젝트의 **"독립 실행형 배포 모델"**과 **"로컬 인터랙티브 유틸리티 도구"**로서의 정체성을 보존하기 위해서는, 서버 렌더링 종속성을 가지는 Next.js 대신 가볍고 배포 유연성이 극대화된 **React + Vite** 스택을 유지하는 것이 가장 합당한 아키텍처 결정입니다.
