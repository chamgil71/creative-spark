## 1. Claude 플랫폼 전체 구조

Claude는 Anthropic이 만든 AI 어시스턴트로, 접근 방식에 따라 세 가지 형태로 나뉜다.

| 형태 | 접근 방법 | 특징 |
|------|-----------|------|
| 웹 Claude | claude.ai 브라우저 접속 | 가장 쉬운 진입, UI 기반 |
| Claude Desktop | 앱 설치 (Mac/Windows) | MCP 서버 연결 지원, 로컬 파일 접근 |
| Claude Code (CLI) | 터미널에서 `claude` 명령어 | 코드 편집·실행 전문, 에이전틱 코딩 |

**플랜 구조**
- **Free**: 하루 제한 메시지, Sonnet 모델
- **Pro** (월 $20): 더 많은 메시지, Opus·Sonnet 선택, 프로젝트 기능, 확장 사고 모드
- **Max** (월 $100/200): 5×~20× 더 많은 사용량, 우선 처리
- **Team/Enterprise**: 조직 단위 관리, 데이터 비학습 보장

## 2. 웹 Claude 기본 사용법

### 2-1. 처음 접속 및 화면 구조

`claude.ai` 접속 후 로그인하면 왼쪽 사이드바와 중앙 채팅창이 보인다.

- **왼쪽 사이드바**: 대화 목록, 즐겨찾기 대화, 프로젝트 목록
- **상단 중앙**: 현재 사용 중인 모델명 표시 (클릭하면 모델 변경 가능)
- **하단 입력창**: 파일 첨부(📎), 음성 입력(🎤), 전송 버튼

### 2-2. 설정 (Settings)

좌측 하단 프로필 아이콘 클릭 → **Settings** 선택

#### 프로필 / 이름 지정
- **Settings → Profile** 탭으로 이동
- `Name` 필드에 Claude가 부를 이름 입력
- `What should Claude know about you?` 칸에 직업, 관심사, 선호하는 응답 방식 입력
- 저장 후 신규 대화부터 즉시 반영됨

#### 기본 지침 지정 (Custom Instructions)
- **Settings → Profile → What would you like Claude to know about you?**: 나에 대한 배경 정보 (직업, 전문 분야, 언어 선호)
- **Settings → Profile → How would you like Claude to respond?**: 응답 스타일 지정 (예: "항상 한국어로 답해줘", "코드는 설명 없이 바로 줘", "짧고 명확하게")
- 이 설정은 모든 대화에 자동 적용됨. 프로젝트 지침과 중복 시 프로젝트 지침이 우선

#### 사용량 보기
- **Settings → Account** 탭 → `Usage` 항목에서 현재 메시지 사용량, 남은 양 확인
- Pro 플랜은 5시간 단위 리셋, Max는 더 넉넉한 풀

#### 커넥터 (Integrations)
- **Settings → Integrations** 탭
- Google Drive, Google Calendar, Gmail 등 외부 서비스 연결
- 연결 후 대화에서 "내 캘린더에서 이번 주 일정 보여줘" 식으로 직접 조회 가능
- 각 커넥터 옆 **Connect** 버튼 클릭 → OAuth 인증 → 권한 허용 → 연결 완료

#### Claude Code 설정
- **Settings → Claude Code** 탭 (Claude Code를 설치한 경우 표시)
- 웹에서 Claude Code 세션 연동 여부, 프로젝트 동기화 설정

### 2-3. 아티팩트 (Artifacts)

대화 중 Claude가 생성한 독립 실행 가능한 결과물. 채팅창 오른쪽에 별도 패널로 표시됨.

**아티팩트로 만들 수 있는 것**
- HTML/CSS/JS 기반 인터랙티브 앱 (간단한 계산기, 게임, 대시보드)
- SVG 그래픽, 차트
- 마크다운 문서
- 코드 스니펫

**사용 방법**
1. 채팅에서 "HTML로 할일 관리 앱 만들어줘" 입력
2. 우측 패널에 미리보기 + 코드가 동시에 표시됨
3. 패널 상단 탭에서 **Preview** / **Code** 전환 가능
4. 우측 상단 **Copy** 버튼으로 코드 복사, **Download** 버튼으로 파일 저장
5. "버튼 색을 파란색으로 바꿔줘" 식으로 후속 수정 요청 가능 → 아티팩트 실시간 업데이트
6. **Publish** 버튼(우측 상단) 클릭 시 공개 URL 생성 가능 (타인 공유 가능)

**활용 예시**
- 간단한 포트폴리오 페이지 뼈대 제작
- 데이터 시각화 차트 (테이블 붙여넣기 → 차트 생성)
- 프레젠테이션용 인터랙티브 슬라이드

### 2-4. 프로젝트 (Projects)

관련 대화를 하나의 공간으로 묶고, 공통 지침과 파일을 공유하는 기능.

**프로젝트 만들기**
1. 좌측 사이드바 상단 **Projects** 섹션 → **+ New Project** 클릭
2. 프로젝트 이름 입력 (예: "마케팅 기획", "개발 프로젝트")
3. 프로젝트 화면으로 이동됨

**프로젝트 지침 설정**
1. 프로젝트 화면 우측 상단 **Edit** 또는 **Project Instructions** 클릭
2. 텍스트 박스에 이 프로젝트 전용 지침 입력
   - 예: "모든 응답은 불릿 포인트 형식으로 정리해줘. 전문 용어는 괄호 안에 한국어 설명을 달아줘."
3. **Save** 클릭 → 이 프로젝트 내 모든 대화에 자동 적용

**파일 업로드**
1. 프로젝트 화면 우측 **+ Add content** 또는 **Upload files** 클릭
2. PDF, Word, 이미지, 코드 파일 등 업로드 가능 (최대 200MB)
3. 업로드된 파일은 프로젝트 내 모든 대화에서 참조 가능
4. "업로드한 보고서를 요약해줘" 식으로 바로 사용

**활용 예시**
- 업무 분야별 프로젝트 분리 (글쓰기 프로젝트, 코딩 프로젝트, 리서치 프로젝트)
- 특정 제품/서비스 관련 문서를 올려두고 반복 질의응답

### 2-5. 스킬 (Skills)

claude.ai **Settings → Integrations → Skills** 탭에서 관리하는 대화 내 슬래시 커맨드 능력 목록. 플러그인을 설치하면 그 플러그인이 제공하는 스킬들이 이 목록에 추가된다.

**사용 방법**
- 대화창에서 `/` 입력 → 현재 활성화된 스킬 목록이 자동완성으로 표시됨
- 원하는 스킬 선택 → 해당 목적에 최적화된 프롬프트 컨텍스트로 대화 시작

**특징**
- 스킬은 단독으로 존재하거나, 플러그인 설치 시 자동 추가됨
- 프로젝트 지침과 달리 매번 설정 없이 즉시 호출 가능
- 현재 활성 스킬 수는 Settings → Integrations → Skills 탭에서 확인

> 스킬을 직접 호출하지 않아도 Claude는 대화 문맥에 따라 자동으로 관련 스킬을 적용하기도 한다.

### 2-6. 커넥터 (Connectors)

claude.ai **Settings → Integrations → Connectors** 탭에서 MCP 기반으로 연결하는 외부 서비스. 웹 Claude에서도 실시간으로 외부 데이터를 조회하거나 서비스를 제어할 수 있다.

**주요 커넥터 예시 (디렉터리에 수십~수백 종이 등록되어 있고 계속 늘어나는 중)**

| 서비스 | 카테고리 | 주요 활용 예 |
|--------|----------|-------------|
| **Slack** | 협업 | 채널 메시지 조회/발송, 스레드 요약 |
| **Notion** | 문서·DB | 페이지 읽기·쓰기, 데이터베이스 조회 |
| **Figma** | 디자인 | 컴포넌트 생성, 디자인 컨텍스트 읽기 |
| **Canva** | 디자인 | 디자인 초안 생성 및 편집 |
| **HubSpot** | CRM | 리드·거래·연락처 조회 및 생성 |
| **Ahrefs** | SEO | 키워드 분석, 경쟁사 백링크 조사 |
| **Amplitude** | 데이터 분석 | 이벤트·퍼널·리텐션 분석 |
| **GitHub** | 개발 | 저장소 조회, PR·이슈 관리 |
| **Jira** | 프로젝트 관리 | 이슈 생성·조회, 스프린트 현황 확인 |
| **Zendesk** | 고객지원 | 티켓 조회, 답변 초안 작성 |
| **Linear** | 이슈 트래킹 | 이슈 생성, 프로젝트 현황 확인 |

> ⚠️ 커넥터 연결 후 Claude는 해당 서비스의 **실제 데이터**에 접근한다. 민감 정보가 포함된 서비스는 연결 범위와 권한을 먼저 확인할 것.

**연결 방법**
1. Settings → **Integrations** → **Connectors** 탭 → 원하는 서비스 **Connect** 클릭
2. OAuth 인증 또는 API 키 입력 → 권한 허용
3. 대화에서 자연어로 바로 사용:
   - `"Slack #general 채널 오늘 메시지 요약해줘"`
   - `"Notion에 오늘 회의록 페이지 만들어줘"`
   - `"Ahrefs에서 'AI 도구' 키워드 분석 결과 보여줘"`

### 2-7. 플러그인 (Plugins)

claude.ai **Settings → Integrations → Plugins** 탭에서 설치하는 목적별 기능 패키지. Anthropic 공식 또는 파트너사가 제공하며, 설치 시 관련 스킬들이 자동으로 추가된다.

**설치 방법**
1. Settings → **Integrations** → **Plugins** 탭
2. 원하는 플러그인 카드 → **Add** 클릭
3. 대화창 `/` 입력 시 해당 플러그인의 스킬 목록이 추가된 것을 확인

**플러그인 예시 — Marketing (Anthropic 공식)**

마케팅 채널 전반의 콘텐츠 제작·캠페인 기획·성과 분석을 지원. 설치 시 8개 스킬이 추가된다.

| 스킬 커맨드 | 기능 |
|------------|------|
| `/brand-review` | 콘텐츠 브랜드 일관성 검토 |
| `/campaign-plan` | 멀티채널 캠페인 기획 |
| `/content-creation` | SEO 최적화 블로그 포스트 작성 |
| `/draft-content` | 콘텐츠 초안 생성 |
| `/email-sequence` | 이메일 시퀀스 작성 |
| `/seo-audit` | 콘텐츠 SEO 감사 및 갭 분석 |
| `/competitive-brief` | 경쟁사 포지셔닝 분석 |
| `/performance-report` | 채널별 성과 리포트 작성 |

> 플러그인은 스킬의 묶음 패키지다. 업무 영역별로 적합한 플러그인을 설치해두면 매번 프롬프트를 새로 작성할 필요 없이 `/커맨드` 하나로 최적화된 작업을 시작할 수 있다.

## 3. Claude Desktop

### 3-1. 설치 및 기본 설정

- `claude.ai/download` 또는 Anthropic 공식 사이트에서 macOS/Windows 설치 파일 다운로드
- 설치 후 실행 시 웹 Claude와 동일한 계정으로 로그인
- 외관은 웹과 유사하지만 **MCP(Model Context Protocol)** 서버 연결이 가능한 것이 핵심 차이

### 3-2. MCP (Model Context Protocol)

Claude Desktop이 외부 서버·도구·로컬 파일 시스템과 통신할 수 있게 해주는 프로토콜.

**MCP 서버 추가 방법**
1. Claude Desktop 상단 메뉴 → **Claude** → **Settings** (Mac: `Cmd+,`)
2. **Developer** 탭 → **Edit Config** 클릭
3. `claude_desktop_config.json` 파일이 열림
4. 아래 형식으로 MCP 서버 추가:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/yourname/Documents"]
    },
    "notion": {
      "command": "npx",
      "args": ["-y", "@notionhq/notion-mcp-server"],
      "env": {
        "NOTION_API_TOKEN": "your_token_here"
      }
    }
  }
}
```

5. 파일 저장 → Claude Desktop 재시작
6. 채팅창 좌측 하단에 연결된 MCP 도구 아이콘 표시됨 (망치 아이콘 🔨)
7. 이제 "내 Documents 폴더에 있는 파일 목록 보여줘" 식으로 로컬 파일 접근 가능

**주요 MCP 서버 목록**
- `@modelcontextprotocol/server-filesystem` : 로컬 파일 읽기/쓰기
- `@modelcontextprotocol/server-github` : GitHub 저장소 조작
- `@notionhq/notion-mcp-server` : Notion 페이지 읽기/쓰기
- `@modelcontextprotocol/server-google-drive` : Google Drive 파일 접근
- `@modelcontextprotocol/server-slack` : Slack 메시지 조회/발송
- `@modelcontextprotocol/server-puppeteer` : 브라우저 자동화

### 3-3. Claude Desktop vs 웹 Claude 차이

| 항목 | 웹 Claude | Claude Desktop |
|------|-----------|----------------|
| MCP 서버 | 불가 | 가능 |
| 로컬 파일 접근 | 불가 | MCP 통해 가능 |
| 오프라인 사용 | 불가 | 불가 (서버 연결 필요) |
| 사용 편의성 | 더 쉬움 | 설정 필요 |
| 커넥터(Google 등) | 가능 | MCP로 대체 |

## 4. Claude Code (CLI)

### 4-1. 개요

터미널에서 실행하는 Claude AI 코딩 도구. 단순 코드 생성이 아니라 파일을 직접 읽고 수정하며, 테스트를 실행하고, Git을 조작하는 **에이전틱(Agentic)** 방식으로 동작한다.

**설치**
```bash
npm install -g @anthropic-ai/claude-code
```

**실행**
```bash
# 프로젝트 디렉토리로 이동 후
claude
```

### 4-2. 기본 사용 방법

터미널에서 `claude` 입력 시 대화형 프롬프트 시작.

```
> 이 프로젝트의 README를 읽고 버그 목록을 정리해줘
> src/app.py의 login 함수에 입력값 검증을 추가해줘
> 방금 수정한 내용을 테스트하고 결과 알려줘
> 변경사항을 git commit해줘
```

- Claude가 파일을 읽고, 코드를 수정하고, 터미널 명령을 실행하는 것을 실시간으로 확인 가능
- 각 단계에서 사용자 승인 요청 (파일 수정, 명령 실행 전 확인)
- `--dangerously-skip-permissions` 플래그로 자동 승인 모드 가능 (주의 필요)

### 4-3. 주요 슬래시 명령어

### 4-4. CLAUDE.md (프로젝트 지침 파일)

## 5. Claude 활용 전략

### 5-1. 지침 작성법

효과적인 지침의 4가지 요소: **역할 + 맥락 + 형식 + 제약**

**역할**: Claude가 맡을 페르소나
> "너는 10년 경력의 UX 라이터야. 기술 문서를 일반인도 쉽게 읽을 수 있도록 바꿔줘."

**맥락**: 배경 정보 제공
> "우리 서비스는 40~60대 직장인을 대상으로 한 건강관리 앱이야. 사용자들은 IT에 익숙하지 않아."

**형식**: 원하는 출력 형태
> "결과는 마크다운 표로 정리하고, 각 항목에는 예시를 하나씩 포함해줘."

**제약**: 하지 말아야 할 것
> "전문 의학 용어는 쓰지 마. 영어는 괄호 안에만 병기해."

**지침 만드는 실전 흐름**
1. 먼저 원하는 작업을 자연어로 질문해보기
2. 결과가 마음에 안 들면 "더 구체적으로/다르게 해줘"로 방향 조정
3. 만족스러운 결과가 나오면 "방금 내가 준 요청을 재사용 가능한 지침으로 정리해줘" 요청
4. 정리된 지침을 프로젝트 지침 또는 CLAUDE.md에 등록

### 5-2. 스킬(Skill) 등록 및 호출

Claude Code에서 반복 작업을 슬래시 명령어로 등록하는 기능.

**스킬 파일 위치**: `.claude/commands/<skill-name>.md`

**예시: 보고서 스킬**

`.claude/commands/report.md` 파일 내용:
```markdown
# 보고서 작성 스킬

입력된 내용을 기반으로 다음 구조의 보고서를 작성한다:

1. 핵심 요약 (3줄 이내)
2. 배경 및 현황
3. 주요 분석
4. 결론 및 제안사항

형식: 마크다운, 제목은 ## 사용, 항목은 불릿 포인트
```

등록 후 사용:
```
> /report 이번 분기 매출 데이터 분석해줘
```

**스킬 파이프라인 예시**
```
/report (초안 생성)
  → /refine (문체 정제)
  → /export-hwpx (HWPX 파일 변환)
```

### 6-1. Claude Cowork (협업 도구 연동)

**Claude + Excel**
- 엑셀 파일을 대화에 첨부 → 데이터 분석, 수식 제안, 피벗 구조 설계
- "이 매출 데이터에서 전월 대비 증감률 구하는 수식 알려줘"
- MCP 연결 시 직접 파일 수정 가능

**Claude + Google Docs / Word**
- 문서 파일 첨부 후 요약, 교정, 번역, 재구성 요청
- Google Drive MCP 연결 시 문서 직접 읽기/쓰기 가능

**폴더 정리 자동화**
- 파일 시스템 MCP 연결 후: "Downloads 폴더를 날짜별/유형별로 자동 정리해줘"
- Python 스크립트를 생성해서 직접 실행하는 방식도 가능

**카카오 메모 연동 (제한적)**
- PlayMCP(카카오 공식 MCP)는 카카오톡 **'나와의 채팅방'에 메모를 보내는 기능만** 제공한다
- 타인과의 대화 조회·발송은 지원하지 않는다 — 자기 자신에게만 메모를 전송할 수 있다
- 톡캘린더·카카오맵·선물하기 등 다른 카카오 서비스용 MCP는 별도로 제공된다
- Claude Desktop Settings → MCP 서버에 카카오 MCP 추가
- "오늘 정리한 할 일을 나와의 채팅방으로 보내줘" 정도가 현실적인 활용 범위

### 6-2. Claude 루틴 (정기 자동화)

Claude Code의 `/schedule` 기능으로 반복 작업을 클라우드 에이전트로 등록.

```
매일 오전 8시:
- 오늘의 뉴스 헤드라인 수집
```

**설정 방법**
1. Claude Code 터미널에서 `/schedule` 입력
2. 작업 내용, 실행 시각(cron 표현식), 반복 주기 지정
3. 클라우드에 등록 후 PC가 꺼져 있어도 자동 실행
4. `/schedule list`로 등록된 루틴 확인, `/schedule delete`로 삭제

## 7. 에이전틱 코딩 도구 비교

에이전틱 코딩: AI가 코드를 생성하는 것을 넘어, 파일을 읽고 수정하며 테스트와 디버깅까지 자율적으로 수행하는 방식.

### 7-1. Claude Code (Anthropic)

- **실행 방식**: 터미널 CLI (`claude` 명령어)
- **강점**: 대용량 컨텍스트(Opus/Sonnet 4.6 기준 표준 요금으로 최대 1M 토큰), 정교한 코드 이해, MCP 생태계
- **특징**: CLAUDE.md 프로젝트 지침, 메모리 시스템, 훅(hook) 자동화
- **비용**: Claude Pro/Max 구독 필요 (월 $20/$100)
- **IDE 통합**: VS Code 익스텐션, JetBrains 플러그인 지원

### 7-2. Gemini CLI / Project IDX (Google)

- **Gemini CLI**: `gemini` 명령어로 터미널에서 사용, Gemini 3 계열(3.1 Pro, 3.5 Flash) 기반
- **Project IDX**: 브라우저 기반 클라우드 IDE에 Gemini 통합
- **강점**: Google 서비스(Drive, Docs, Sheets) 네이티브 연동, 무료 사용량 넉넉
- **특징**: Gemini 3.5 Flash로 빠른 응답, 멀티모달(이미지·코드 동시 처리)

### 7-3. OpenAI Codex / ChatGPT (OpenAI)

- **Codex CLI**: 터미널에서 `codex` 명령어로 실행
- **ChatGPT Canvas**: 웹 ChatGPT 내 코드 편집 전용 패널
- **강점**: GPT-5.5 기반, 광범위한 언어/프레임워크 지원
- **특징**: 코드 실행 환경 내장(Python), 인터프리터 기능

### 7-4. GitHub Copilot

- **설치**: VS Code 익스텐션 설치 후 GitHub 계정 연동
- **동작 방식**: 코드 작성 중 자동 완성 제안 (Tab으로 수락)
- **Copilot Chat**: 사이드바 채팅창에서 코드 설명·수정 요청
- **Copilot Workspace**: GitHub 이슈→코드 변경→PR 자동화
- **비용**: 개인 월 $10, 비즈니스 월 $19/인

### 7-5. Cursor

- **형태**: VS Code 기반 독자적 AI IDE (별도 앱 설치)
- **동작 방식**: `Ctrl+K`로 인라인 편집, `Ctrl+L`로 채팅 사이드바
- **강점**: 전체 코드베이스 인덱싱, 파일 간 참조 이해도 높음
- **Composer 모드**: 멀티 파일 동시 수정 (에이전틱 방식)
- **비용**: 무료(제한), Pro 월 $20

```
    ↓ GitHub Actions 워크플로우 실행

    ↓
[Wiki 사이트] 자동 업데이트된 웹사이트
```

- Obsidian에서 글 저장 → Git 자동 commit → GitHub에 push → Vercel이 감지해 자동 빌드
- 도구: Obsidian Git 플러그인, GitHub, Vercel (Next.js 또는 Hugo/Jekyll)
- 비용: Obsidian 무료, GitHub 무료, Vercel 무료 플랜

### 10-2. 시황 뉴스 자동화 파이프라인

```
[Claude 루틴] 매일 오전 7시 실행
    ↓
[뉴스 수집] 주요 경제/증시 뉴스 크롤링
    ↓
[GitHub Actions] 워크플로우 트리거
    ↓
[Claude API] 뉴스 요약 및 시황 분석
    ↓
[다중 발송]
  ├── Gmail로 이메일 발송
  ├── 텔레그램 Bot으로 메시지 발송
  └── SNS (X/블로그) 자동 포스팅
```

- 핵심: GitHub Actions의 cron 스케줄로 정기 실행
- `.github/workflows/daily-news.yml` 에 `schedule: cron: '0 22 * * *'` (UTC 기준)

### 10-3. AI 주간 뉴스 분석 레포트

```
[GitHub Actions] 매주 월요일 실행
    ↓
[Gemini API] 지난 주 뉴스 수집 및 분류
    ↓
[Claude API] 주제별 심층 분석 및 인사이트 작성
    ↓
[레포트 생성] 마크다운 또는 HTML 형식
    ↓
[Gmail] 구독자에게 주간 뉴스레터 발송
```

- Gemini: 검색/수집 역할 (Google 검색 통합 강점)
- Claude: 분석/작성 역할 (긴 문서 이해·생성 강점)
- 두 AI를 역할 분리해서 조합하는 패턴

### 10-4. 문서→슬라이드 자동화 파이프라인

```
[Obsidian/텍스트 메모]
    ↓ Claude Code 처리
[마크다운 문서]
    ↓ 변환 스크립트
```

- 전체 파이프라인을 GitHub Actions로 자동화 가능

## 11. AI 협업 레슨런 — 실전에서 건진 14가지 원칙

AI 도구를 한동안 실제 업무에 적용하다 보면 "이걸 처음부터 알았더라면"이라는 깨달음들이 쌓인다. 아래는 시행착오 끝에 정리한 14가지 실전 원칙이다.

### 11-1. 역할 분리 — AI에게 설계를 위임하지 마라

AI는 구현을 잘한다. 그러나 **무엇을 만들지 결정하는 일**은 사람의 몫이다. "알아서 좋은 구조로 짜줘"라고 요청하면 AI는 그럴듯한 결과를 내놓지만, 그 설계가 프로젝트의 맥락에 맞는지는 검증되지 않는다. 역할 분리 원칙: *설계는 사람이, 구현은 AI가*.

### 11-2. 실패 기록 — 오류를 자산으로 만들어라

잘 된 결과만 저장하는 것이 당연해 보이지만, **실패한 시도와 오류 메시지도 기록**해야 한다. AI는 이전 대화를 기억하지 못한다. 오류 로그와 "이걸 시도했는데 안 됐다"는 기록이 다음 세션의 출발점이 된다.

### 11-3. 파이프라인 설계 — 인터페이스를 계약서처럼 명시하라

여러 도구를 연결할 때 각 단계의 입출력 형식을 먼저 정의한다. "마크다운 파일 → 스크립트 → HTML"처럼 **인터페이스를 계약**으로 취급하면, 중간 단계가 바뀌어도 전체가 무너지지 않는다.

### 11-4. 설정·코드 분리 — 변하는 것과 변하지 않는 것을 구별하라

- **변하는 것** (스타일, 색상, 문구, API 엔드포인트) → 설정 파일(JSON/YAML/env)
- **변하지 않는 것** (파싱 로직, 렌더링 엔진, 데이터 구조) → 코드

이 경계를 처음부터 지키면 AI에게 "설정 파일만 바꿔줘"라는 정확한 지시가 가능해진다.

### 11-5. 원본 보존 — 복사→검증→삭제 순서를 지켜라

AI가 파일을 수정하기 전, 반드시 원본을 복사한다. 수정 결과를 검증한 뒤에야 원본을 삭제한다. "AI가 알아서 백업하겠지"는 착각이다.

### 11-6. 맥락 명시 — 세 줄로 독자, 형식, 금지를 적어라

AI에게 작업을 맡길 때 최소 세 가지를 명시한다.

> "**독자:** 비개발자 팀장  
> **형식:** 3줄 이내 요약 후 표  
> **금지:** 기술 약어 사용 금지"

이 세 줄만 있어도 결과물의 품질이 크게 달라진다.

### 11-7. 자산 축적 — 대화는 흘러가지만 파일은 남는다

대화창에서 AI와 나눈 좋은 프롬프트, 성공한 패턴, 유용한 지시 방식은 반드시 **SKILL.md 파일에 저장**해야 한다. 대화는 세션이 끝나면 사라지지만, 파일은 다음 프로젝트에도 재사용된다.

### 11-8. 고통 우선 — 자동화는 고통이 명확할 때 시작하라

"이걸 자동화하면 좋겠다"는 막연한 생각으로 파이프라인을 만들면 쓸모없는 자동화가 된다. **반복하면서 실제로 고통스러운 부분**을 먼저 자동화하라. 고통이 명확할수록 자동화의 범위도 명확해진다.

### 11-9. 최소 인프라 — 가장 작은 것으로 시작하라

첫 시도에 완벽한 시스템을 구축하려 하면 완성 전에 포기하게 된다. **가장 작은 v1**으로 시작해 실제로 동작하는 것을 확인한 뒤 점진적으로 확장한다.

### 11-10. 미완성 기록 — 완성보다 기록이 먼저다

미완성 상태라도 현재까지 한 작업, 막힌 지점, 다음에 할 일을 기록해 두면 재개가 쉬워진다. 완성된 결과물보다 **진행 과정의 기록**이 더 가치 있는 경우가 많다.

### 11-11. 프롬프트 밀도 — 네 가지를 명시하라

효과적인 프롬프트에는 네 가지가 들어간다.

| 항목 | 설명 |
|------|------|
| **현상** | 현재 어떤 상황인가 |
| **예상** | 원래 어떻게 되어야 하는가 |
| **실제** | 실제로 어떻게 되고 있는가 |
| **시도** | 이미 해본 것은 무엇인가 |

이 네 가지가 갖춰진 프롬프트는 AI가 핵심을 파악하는 시간을 크게 줄인다.

### 11-12. 도구 연결 — n8n → Ollama → Resend → Notion

개별 도구의 성능보다 **연결 방식**이 결과를 결정한다. 트리거(n8n/웹훅) → 처리(Claude/Ollama) → 전달(Resend/Slack) → 저장(Notion/DB) 형태의 파이프라인을 구성하면, 단일 도구로는 불가능한 자동화가 가능해진다.

### 11-13. 즉시 시작 — 완벽한 설계를 기다리는 것이 가장 비싼 실수

완벽한 설계가 나올 때까지 시작하지 않는 것은 **시간 낭비 중 가장 비싸다**. 불완전하더라도 v1을 만들어 돌려봐야 어디서 실제 문제가 생기는지 알 수 있다. 설계는 실행하면서 고쳐나가는 것이다.

### 11-14. 형식 명시 — 몇 줄, 어떤 요소, 누가 읽는가

출력 형식을 명시하지 않으면 AI는 자신이 편한 형식으로 답한다. 항상 "**3줄 이내로, 표 포함, 임원이 읽는 보고서**"처럼 형식 조건을 먼저 정의하고 작업을 맡긴다.

## 12. 설계와 구조화 — 지침이 생산성을 결정한다

### 12-1. 단편적 질답의 한계

AI를 처음 쓰는 방식은 대부분 단편적 질문과 답변이다. "이 코드 고쳐줘", "이 문장 요약해줘". 이 방식은 빠르지만, 축적되지 않는다. 매번 맥락을 다시 설명해야 하고, 이전 대화의 결과를 다음 대화에서 활용하기 어렵다.

단편적인 대화가 반복될수록 **AI에게 설명하는 시간이 전체 작업의 절반**을 차지하게 된다.

### 12-2. 구조화된 재사용성 — SKILL.md의 힘

반복하는 작업 패턴을 파일로 만들면 재사용이 가능해진다.

```
SKILL.md 구성 예시
---
[스킬 이름] 주간 업무 보고서 생성

[언제 사용] 매주 금요일 업무 정리 시

[입력] 이번 주 완료한 업무 목록 (불릿 형식)

[출력 형식]
- 제목: "YYYY-WW 주간 업무 보고"
- 섹션: 완료/진행중/계획
- 길이: A4 1장 이내
- 어조: 공식 보고서 (존댓말)

[예시 입력/출력]
... (실제 예시 포함)
```

이 파일을 Claude 프로젝트에 업로드해 두면, "주간 보고서 작성해줘"라는 한 문장만으로 매번 동일한 품질의 결과를 얻는다.

### 12-3. 설계(지침)의 중요성 — 무엇을 만들지가 어떻게 만들지보다 중요하다

AI 도구가 강력해질수록 **"무엇을 만들지"를 결정하는 능력**이 더 중요해진다. 잘못된 방향으로 아무리 빠르게 달려도 원하는 곳에 도착할 수 없다.

좋은 지침서(SKILL.md, 시스템 프롬프트, 프로젝트 설정)를 작성하는 능력이 곧 AI 활용 능력이다. 지침이 정밀할수록:

- AI가 맥락을 파악하는 시간이 줄어든다
- 반복 수정 횟수가 줄어든다
- 다른 팀원도 같은 품질을 재현할 수 있다
- 프로젝트가 바뀌어도 지침만 수정하면 된다

단편적인 "프롬프트 꿀팁"을 모으는 것보다, 자신의 업무에 맞는 **구조화된 지침 체계**를 만드는 것이 장기적으로 훨씬 더 큰 생산성 차이를 만든다.

## 13. 서비스 연결과 조립 — 도구는 조합될수록 강해진다

### 13-1. 단일 도구의 한계

Claude는 문서를 잘 쓴다. Gemini는 검색을 잘 한다. n8n은 자동화를 잘 한다. 그러나 각각을 따로 쓰는 것과 **연결해서 쓰는 것**은 결과가 다르다.

조합의 원칙: 각 도구가 가장 잘 하는 한 가지만 맡기고, 그 결과를 다음 도구로 넘긴다.

### 13-2. 연결 방식 — API, MCP, 웹훅

| 연결 방식 | 특징 | 적합한 상황 |
|-----------|------|-------------|
| **API 직접 호출** | 완전한 제어, 코드 필요 | 개발자, 커스텀 파이프라인 |
| **MCP 서버** | Claude와 서비스 직접 연동 | Claude Code 환경 |
| **웹훅** | 이벤트 기반 자동 트리거 | n8n, Zapier, Make 연동 |
| **노코드 플랫폼** | 시각적 구성, 코드 불필요 | 비개발자, 빠른 프로토타이핑 |

### 13-3. 실전 파이프라인 예시

**문서 자동 배포 파이프라인**

```
[Obsidian 노트 저장]
    ↓ 웹훅 트리거 (파일 변경 감지)
[n8n 워크플로우 시작]
    ↓ Claude API 호출
[마크다운 → 구조화 문서 변환]
    ↓ 빌드 스크립트 실행
[HTML 슬라이드 / PPTX 생성]
    ↓ Notion API
[팀 공유 페이지 자동 업데이트]
    ↓ Slack 웹훅
[담당자에게 완료 알림 전송]
```

**이메일 분류 및 요약 파이프라인**

```
[새 이메일 수신 (Gmail)]
    ↓ n8n 트리거
[Claude API: 분류 + 요약]
    ↓ 분류 결과에 따라 분기
[긴급] → Slack 즉시 알림
[일반] → Notion DB 저장
[광고] → 자동 아카이브
```

### 13-4. 조립의 핵심 원칙

1. **가장 고통스러운 지점부터** 자동화를 시작한다
2. **각 단계의 입출력 형식**을 먼저 정의한다 (인터페이스 계약)
3. **단계별로 검증**하고, 전체를 한꺼번에 연결하지 않는다
4. **실패 시 알림**을 반드시 설정한다 (자동화가 조용히 실패하는 것이 최악)
5. **로그를 남긴다** — 어느 단계에서 문제가 생겼는지 추적 가능해야 한다

서비스를 연결하고 조립하는 능력이 갖춰지면, AI 도구 하나의 성능보다 **파이프라인의 설계**가 훨씬 더 중요해진다는 것을 실감하게 된다.

## 결론 — 직접 해봐야 안다

AI 도구에 대한 글은 넘쳐난다. 기능 설명, 비교표, 활용법, 프롬프트 팁. 이 가이드도 그 중 하나다.

그러나 어떤 설명을 읽는 것만으로는 실제 감각이 생기지 않는다. 백 번 읽는 것보다 한 번 써보는 것이 낫고, 백 번 써보는 것보다 한 번 제대로 물어보는 것이 낫다.

"어떤 도구가 제일 좋아요?"가 아니라, "내 업무에서 가장 반복적이고 고통스러운 부분이 무엇인가?"를 묻는 것에서 시작해야 한다. 그 답이 나왔을 때, 이 가이드에서 설명한 도구와 원칙들이 비로소 의미를 가진다.

백문이 불여일견, 백견이 불여일타, **백타가 불여일문** !!
