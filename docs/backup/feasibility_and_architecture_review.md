# 프로젝트 타당성 및 아키텍처 냉철한 검토 보고서

본 보고서는 "포맷-디자인 교차 변환 프레임워크"가 쓸데없이 무거운 시스템이 되지 않기 위해 준수해야 할 경계 조건, 유사 서비스 및 대안과의 비교 분석, 그리고 프레임워크의 한계점과 이에 따른 프로젝트 구조 개편안을 냉철한 관점에서 다룹니다.

---

## 1. 유사 서비스/도구 대조 및 본 프로젝트의 독보적 가치

기존의 마크다운 기반 문서화 및 슬라이드 변환 도구들과 비교했을 때, 본 프로젝트가 왜 가볍게 유지되어야 하며 굳이 별도로 개발해야 하는지에 대한 타당성 분석입니다.

| 도구명 | 지원 형식 | 주요 장점 | 한계점 및 본 프로젝트의 차별성 |
| :--- | :--- | :--- | :--- |
| **Marp** | MD ──> HTML / PPTX | - 프레젠테이션 특화 파이프라인 표준<br>- 뛰어난 성능과 풍부한 CSS 테마 | - **PPTX 변환 시 전체 슬라이드를 통이미지(SVG/PNG)로 구워버림** (텍스트 및 표의 사후 편집이 완전 불가능).<br>- *차별성*: 본 프로젝트는 `pptxgenjs`를 이용해 실제 PPTX 파일 내부에 편집 가능한 **벡터 텍스트 및 테이블 객체**로 컴파일함. |
| **Pandoc** | 다중 포맷 간 변환 | - 문서 변환계의 스위스 아미 나이프 | - 레이아웃 구조가 평평하게 렌더링되며, 숏코드(`:::`)를 통한 복잡한 시각 컴포넌트(그리드, 에디터 목업, 터미널 등)를 다층 레이아웃으로 변환 불가.<br>- *차별성*: 본 엔진은 29종의 특화 숏코드를 직접 해독하여 완벽한 UI 레이아웃을 HTML/PPTX로 보존. |
| **Slidev** | MD ──> HTML Slide | - 개발자 친화적인 화려한 웹 슬라이드 쇼 | - 실제 오프라인 다운로드가 가능한 MS 파워포인트(`.pptx`) 파일 포맷 바이너리 빌드를 공식적으로 미지원.<br>- *차별성*: 로컬 파일 쓰기 API 및 바이너리 직접 생성 지원. |

### 🔍 "쓸데없이 무거운 시스템이 되는가?"에 대한 냉철한 진단
* **위험 요인**: Puppeteer, Playwright 등의 브라우저 제어 라이브러리를 동원해 HTML 화면을 스크린샷 캡처하여 PDF나 PPTX로 얹는 방식을 채택하는 순간, 노드 모듈 크기가 수백 MB로 폭증하고 컴파일 시간이 10배 이상 지연됩니다.
* **경량화 조건**: 본 프로젝트는 무거운 브라우저 가상 기동을 지양하고, **순수 JS/정규식 파서**와 **`pptxgenjs` 메모리 렌더링**만으로 컴파일을 완결 지어야 합니다. 가벼운 빌드 스택을 유지하기 때문에 1초 이내에 저장과 생성이 즉각 끝나는 응답성을 달성할 수 있습니다.

## 2. 한계점 및 제약 사항 (Limitations)

1. **마크다운 숏코드 규격 종속성**:
   * 본 프레임워크가 2단 그리드나 터미널 목업을 변환해 내는 비결은 독창적인 `:::` 숏코드 문법입니다. 이로 인해 타 플랫폼(예: 일반 Github Markdown 뷰어)에서 원본 MD를 볼 때 문법이 깨져 보이는 종속성 문제가 발생합니다.
2. **바이너리 PPTX 변환의 디자인 왜곡**:
   * HTML/CSS에서는 `flex`나 `grid`를 통해 반응형으로 요소 크기가 동적 조절되나, PPTX는 절대 좌표(인치 단위)계로만 동작합니다. 이로 인해 숏코드 내 텍스트 양이 많아질 경우 PPTX 내부에서 글자가 셰이프 바깥으로 오버플로우되거나 겹치는 물리적 한계가 상존합니다.

## 3. 프로젝트 디렉토리 구조 재정리안

기능이 무분별하게 팽창하지 않고 파트별(컴파일러, 디자인 지침, 배포 환경) 독립성을 갖도록 아키텍처 구조를 다음과 같이 재정리할 것을 제안합니다.

```
creative-spark/
├── src/                        # 프론트엔드 리액트 에디터 스택 (UI 계층)
├── config/                     # 디자인 토큰 저장소 (Data 계층)
│   ├── styles.json             # 글로벌 컬러/폰트 테마 규격
│   ├── pptdesign.config.json   # PPTX 렌더링 물리적 규격
│   └── htmldesign.config.json  # HTML 렌더링 활성 스타일 규격
├── scripts/                    # 컴파일 빌드 핵심 파이프라인 (Engine 계층)
│   ├── build-guide.mjs         # MD ──> 가이드 HTML 컴파일러
│   ├── build-presentation.mjs  # MD ──> 슬라이드 HTML 컴파일러
│   ├── md-to-pptx.mjs          # MD ──> PPTX 바이너리 컴파일러
│   ├── designspec-md-to-html.mjs # 디자인 지침 ──> 스타일북 컴파일러 (Node.js)
│   └── pptx-renderers/         # PPTX 숏코드별 독립 렌더러 모듈
├── storage/                    # 데이터 보관소
│   └── designspec/             # 디자인 시스템 가이드라인 리포지토리
└── docs/                       # 기획 및 아키텍처 문서 저장소
    ├── plan/                   # 미래 개발 계획 및 타당성 분석서
    └── plan_Impl/              # [신설] 현재 가동 중인 시스템의 상세 아키텍처 명세서 (재구성용)
```

이 정리 구조를 준수함으로써, 핵심 빌더 엔진(`scripts/`)과 디자인 명세 저장소(`storage/`), 그리고 기획 지침(`docs/`)의 경계가 명확해져 유지보수가 용이해집니다.
